# Задача: новый раздел «Настройки» — вынести «Мониторинг» из сайдбара + разделы из ТЗ (Vue 3 + Laravel 10)

## Контекст
В сайдбаре админки был отдельный блок «МОНИТОРИНГ» (статусы «Очереди» и «WS» в виде цветных точек), торчащий отдельной секцией внизу списка разделов. Нужно:

1. Убрать блок «МОНИТОРИНГ» из сайдбара как отдельную секцию.
2. Добавить в сайдбар (в конец списка, после «Push-уведомления») один пункт **«Настройки»** (иконка — шестерёнка).
3. Реализовать новый раздел «Настройки», куда переехал мониторинг очередей/WS, плюс — после разбора присланного ТЗ — ещё несколько блоков, для которых в старой админке не было выделенного места: права роли «Менеджер» (ТЗ п.2.3, 13.4), справочники причин отклонения объявлений (ТЗ п.5.5) и причин жалоб (ТЗ п.8.3), переключение языков (ТЗ п.14), состояние SMS-шлюза (ТЗ п.3.2).

Дизайн-референс — секция **7a** в `Admin UI.dc.html` (турн 7, «Настройки»). Сайдбар с обновлённым пунктом «Настройки» — во всех секциях файла (6a/5a/4a/3a/2a/2b/1a/1b/1c показывают тот же обновлённый сайдбар).

Перед началом изучи существующий код (роуты, контроллеры, миграции, Vue-компоненты сайдбара) — не изобретай новый подход к роутингу/авторизации, следуй паттернам проекта.

## Сайдбар
- Удалить секцию-заголовок «МОНИТОРИНГ» и её два пункта («Очереди», «WS») как самостоятельный список в нижней части сайдбара.
- Добавить один пункт навигации «Настройки» сразу после «Push-уведомления», без отдельного заголовка секции (визуально — на одном уровне с «Тарифы» / «Статистика» / «Push-уведомления» в группе «СИСТЕМА»).
- Пункт ведёт на новый роут раздела настроек, подсвечивается активным (заливка акцентным цветом), когда открыт этот раздел — так же, как остальные пункты сайдбара.

## Backend

### Мониторинг (карточки «Очереди» и «WS»)
Не самостоятельная сущность — это дисплей текущего состояния инфраструктуры:
- «Очереди»: подключить к тому, что уже используется в проекте для очередей (Laravel Queue + Horizon, если он есть — читать метрики через `Horizon::` API/Redis; если Horizon не используется — считать `pending`/`failed` напрямую по таблицам `jobs`/`failed_jobs`). Отдавать: количество задач в очереди, количество упавших (`failed_jobs`), имя воркера/очереди, время последнего обновления.
- «WS»: подключить к используемому в проекте websocket-серверу (Laravel Reverb / Soketi / Pusher) — отдавать количество активных соединений и аптайм, если провайдер это предоставляет API; если нет прямого API статистики — минимум отдавать «подключено/не подключено» через healthcheck-пинг сервиса.
- Эндпоинт `GET /admin/settings/monitoring` — отдаёт JSON с обоими блоками, фронт поллит его (например раз в 10–15 сек) для авто-обновления карточек.

### Роли и права
Роли «Администратор» и «Менеджер» уже описаны в ТЗ (п.2.2–2.3, 13.4). Жёстко зафиксированные ограничения менеджера (удаление данных, создание админов/менеджеров, смена ролей, критические системные настройки) реализуются как **проверки на backend** (middleware/policy), а не просто скрытие кнопок на фронте — экран настроек лишь **отображает** эту матрицу как справку (эти права не редактируются).

Единственное реально переключаемое право — доступ менеджера к разделу «Новости» («работа с новостями, если права выданы отдельно», ТЗ п.2.3). Добавить:
- Колонку `can_manage_news` (boolean, default `false`) в таблицу `admins`/`managers` (или отдельную таблицу `manager_permissions`, если в проекте уже есть система прав/пермишенов — тогда завести пермишен `news.manage` там, не создавать параллельную систему).
- Эндпоинт `PATCH /admin/settings/manager-permissions` — переключает это право (нужна проверка, что менять может только «Администратор»).
- Middleware/policy на роуты раздела «Новости» должны проверять именно этот флаг для роли «Менеджер».

### Справочники причин
Две независимые таблицы:
- `rejection_reasons`: `id`, `text_ru`, `text_tk` (nullable), `sort_order`, `is_active` (default `true`), timestamps. Используется при отклонении объявления модератором (ТЗ п.5.5 — «администратор выбирает причину из списка»).
- `complaint_reasons`: `id`, `text_ru`, `text_tk` (nullable), `sort_order`, `is_active`, timestamps. Используется при подаче жалобы пользователем (ТЗ п.8.3 — «причина жалобы выбирается из списка»).

CRUD-эндпоинты для обеих (`GET/POST/PATCH/DELETE`), без мягкого удаления — если причина уже была использована в существующих записях модерации/жалоб, хранить исторически выбранный текст в самой записи объявления/жалобы (не только `reason_id`), чтобы удаление причины из справочника не ломало историю.

### Локализация
Два раздельных значения (не одно и то же!):
- Язык интерфейса самой админки для текущего администратора — если в проекте уже есть per-user настройка локали, использовать её; если нет — хранить в `admins.locale` (`ru`/`tk`).
- Язык приложения по умолчанию для новых пользователей мобильного приложения — глобальная настройка, хранить как `app_settings` (или аналогичная таблица key-value, если она уже есть в проекте) с ключом `default_app_locale`.
- Оба языка (`ru`, `tk`) всегда доступны как варианты — выключать язык целиком нельзя, это переключатели значения по умолчанию, а не вкл/выкл поддержки языка.

### SMS-шлюз
Согласно ТЗ (п.3.2), SMS отправляются через локальный модем/телефон, без сторонних SMS-сервисов. На экране настроек — только **статус** этого локального сервиса, не создание нового способа отправки:
- `GET /admin/settings/sms-gateway` — отдаёт статус подключения (подключено/отключено), метку устройства (модем/телефон), время последней синхронизации.
- `POST /admin/settings/sms-gateway/test` — отправляет тестовое SMS через тот же локальный сервис, что используется для кода подтверждения (ТЗ п.3.2), на номер администратора или на явно указанный тестовый номер; возвращает результат (доставлено/ошибка).

## UI/UX (страница «Настройки»)
Вертикальный список секций, каждая — заголовок-лейбл + один или два блока-карточки в ряд (сетка 1fr 1fr на десктопе, в колонку на меньших экранах):

1. **Мониторинг** — две карточки («Очереди», «WS»): цветная точка-индикатор (красный/жёлтый — проблема, зелёный — ок) + бейдж статуса справа, ниже — два укрупнённых числа-метрики с подписями, внизу — служебная строка (имя воркера/кластера + «обновлено N сек назад»). Автообновление по поллингу.
2. **Роли и права доступа** — таблица (право / Администратор / Менеджер), большинство строк — просто отображение (галочка = разрешено, иконка замка = запрещено и не редактируется), кроме строки «Управление новостями» у менеджера — там переключатель (тумблер).
3. **Справочники причин** — две карточки рядом («Причины отклонения объявлений», «Причины жалоб»): список строк с текстом + иконки редактировать/удалить у каждой, «+» в шапке карточки открывает инлайн-форму добавления новой причины (текстовое поле + подтвердить/отмена).
4. **Локализация и SMS** — карточка «Локализация» (два переключателя-сегмента Русский/Türkmen — один для языка админки, второй для языка приложения по умолчанию) и карточка «SMS-шлюз» (статус-бейдж, описание устройства, кнопка «Отправить тестовое SMS» с состоянием загрузки/результата).

## Стиль
Тёмная тема, консистентная с остальной админкой: карточки `--card-bg` на фоне `--content-bg`, акцент `#6D63F2`/`#4F46E5`, скругления 8–14px, шрифт Golos Text, статус-бейджи — оттенок фона + текст того же цвета (зелёный `--ok`/`--ok-tint` для «ок», красный `--danger`/`--danger-tint` для проблем). Используй существующие в проекте цветовые токены темы вместо новых значений. Полный визуальный референс — секция 7a в `Admin UI.dc.html`.

## Что важно не сломать
- Права менеджера (удаление, создание админов/менеджеров, смена ролей, критические настройки) должны быть закрыты на уровне backend-политик независимо от того, что показывает этот экран — экран лишь справочно отображает матрицу.
- Удаление причины отклонения/жалобы из справочника не должно портить уже существующие записи модерации/жалоб, ссылающиеся на неё (см. раздел про справочники выше).
- Смена языка приложения по умолчанию не должна задним числом менять язык уже зарегistrированных пользователей — влияет только на новых.
- Тестовая отправка SMS должна использовать тот же локальный шлюз, что и продовая отправка кодов подтверждения — не заводить отдельный тестовый канал.
