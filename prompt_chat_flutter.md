# Задача: экран «Чат с поддержкой» (Flutter)

## Контекст

Мобильное приложение — доска объявлений. У пользователя есть один-единственный диалог с поддержкой (администратором) — не чат между пользователями, обращений/тикетов несколько быть не может, только один непрерывный диалог. Backend (Laravel + Sanctum + Laravel Reverb) уже полностью реализован, задеплоен и протестирован — эта задача **только про мобильное приложение**, бэкенд не трогать и не запрашивать в нём изменений, ниже — весь контракт, который нужен.

Функционально нужно (это финальные требования, ничего сверх этого не добавлять):
1. Отправка и получение сообщений.
2. История переписки при открытии экрана.
3. Статус прочтения (сообщения администратора помечаются прочитанными, когда пользователь открыл экран чата).
4. Отображение новых сообщений в реальном времени, без ручного обновления/повторного открытия экрана.
5. Экран однонаправленный для пользователя — ответ со стороны администратора приходит из отдельной админ-панели (не часть этой задачи).

Перед началом изучи существующую структуру проекта (навигация, HTTP-клиент, state-management, дизайн-система, работа с токеном авторизации) и следуй уже принятым в проекте паттернам — не вводи новый HTTP-клиент/стейт-менеджер/навигацию, если в проекте уже есть свой.

## Авторизация (уже реализована в приложении)

Токен получается через существующий у вас flow SMS-логина (`POST /api/v1/auth/verify`) и передаётся во всех запросах ниже заголовком:
```
Authorization: Bearer {sanctum_token}
```
Если токен отсутствует/невалиден — сервер вернёт `401 {"message": "..."}`.

## 1. REST-контракт (зафиксирован, протестирован — не переизобретать)

Базовый URL: `{base_url}/api/v1` (сам `base_url` уже настроен в проекте под dev/прод).

### `GET /chat` — история диалога

Ответ `200`:
```json
{
  "data": [
    {
      "id": 12,
      "sender": "user",
      "text": "Здравствуйте, у меня вопрос по объявлению",
      "is_read": true,
      "created_at": "2026-07-08T11:35:54.000000Z"
    },
    {
      "id": 13,
      "sender": "admin",
      "text": "Добрый день! Слушаем вас",
      "is_read": false,
      "created_at": "2026-07-08T11:40:12.000000Z"
    }
  ]
}
```
- Массив отсортирован по возрастанию `created_at` (старые сверху, новые снизу — обычный чат).
- **Пагинации нет** — отдаётся весь диалог целиком. Если переписка становится очень длинной, это ограничение на будущее (при необходимости — отдельная задача на бэкенд), сейчас рендерить весь список как есть.
- `sender` — только `"user"` или `"admin"`. Свои сообщения — `sender == "user"`, ответы поддержки — `sender == "admin"`.
- `is_read`: для `sender == "user"` — прочитано ли администратором; для `sender == "admin"` — прочитано ли пользователем (см. `PATCH /chat/read` ниже).

### `POST /chat` — отправить сообщение

Body:
```json
{ "text": "Текст сообщения" }
```
- `text` — обязательное, строка, максимум **5000 символов** (это же ограничение продублировать на клиенте: не давать отправить пустую/только-пробельную строку и превышение лимита).

Успех — `201`:
```json
{
  "data": {
    "id": 14,
    "sender": "user",
    "text": "Текст сообщения",
    "is_read": false,
    "created_at": "2026-07-08T12:00:00.000000Z"
  },
  "message": "Сообщение отправлено"
}
```
Ошибка валидации — `422`:
```json
{ "message": "...", "errors": { "text": ["..."] } }
```

### `PATCH /chat/read` — отметить ответы поддержки прочитанными

Без тела. Вызывать **при каждом открытии экрана чата** (и опционально — когда экран уже открыт и по WebSocket пришло новое сообщение от `admin`, чтобы не копилась непрочитанная переписка, пока пользователь реально смотрит на экран).

Ответ `200`: `{"message": "Отмечено как прочитанное"}`.

## 2. Реалtайм — WebSocket (Laravel Reverb, протокол Pusher)

Reverb говорит по протоколу **Pusher**. Для Flutter — пакет **`pusher_channels_flutter`** (pub.dev) или любой другой Pusher-совместимый клиент. Готового Flutter-пакета под «Laravel Echo» нет — Echo это браузерная JS-библиотека, к мобильному приложению отношения не имеет.

Параметры подключения (получить актуальные значения `host`/`port`/`scheme`/`appKey` для текущего окружения у бэкенд-команды — они отличаются между dev/prod; ниже описана сама механика, она не меняется):

```
appKey : публичный ключ Reverb-приложения (безопасно хранить в коде — это НЕ секрет)
host   : хост Reverb-сервера
port   : порт Reverb-сервера (в проде обычно 443 за TLS-прокси)
scheme : ws (dev, без TLS) / wss (прод, обязательно)
```
**Никогда не встраивайте в приложение `REVERB_APP_SECRET`** — этим секретом бэкенд только подписывает ответы авторизации канала, клиенту он не нужен и не должен быть доступен.

### Канал и событие

- Приватный канал: **`private-chat.{user_id}`**, где `{user_id}` — id текущего пользователя (берётся из его собственного профиля, `GET /api/v1/profile`). Подписаться можно только на канал своего собственного `user_id` — сервер проверяет это и отклоняет чужой канал 403-м.
- Слушать событие с именем **`new-message`** (именно так, без точки в начале — `.new-message` это Laravel-Echo-специфичный синтаксис для браузера, к чистому Pusher-клиенту не относится).
- Payload события — та же структура сообщения, что и в REST (`id`, `user_id`, `text`, `sender`, `is_read`, `created_at`).

### Авторизация приватного канала (обязательный шаг, без него подписка не пройдёт)

Приватные каналы Pusher/Reverb требуют HTTP-подтверждение права на подписку. Эндпоинт (уже реализован и протестирован на бэкенде, отдельно от обычного `/api/v1/...` префикса):

```
POST {base_url без /v1}/broadcasting/auth
Headers:
  Authorization: Bearer {sanctum_token}
  Accept: application/json
Body (form-urlencoded, подставляется автоматически клиентской библиотекой):
  socket_id=<из текущего WS-соединения>
  channel_name=private-chat.{user_id}
```
Ответ `200`: `{"auth": "app_key:подпись"}` — этот JSON нужно вернуть библиотеке как есть.
Если `user_id` не совпадает с текущим авторизованным пользователем — `403`.

В `pusher_channels_flutter` это настраивается через кастомный authorizer (в разных версиях пакета может называться иначе — сверьтесь с актуальным README установленной версии): вместо стандартного `authEndpoint` с автоматическими куками нужен колбэк, который сам делает POST на `/broadcasting/auth` с заголовком `Authorization: Bearer` (кука/сессия тут не работают — только Bearer-токен) и возвращает разобранный JSON.

### Надёжность

- При открытии экрана чата: сразу `GET /chat` (синхронизация истории) + подписка на канал. WebSocket — только для мгновенных апдейтов поверх уже загруженной истории, не единственный источник правды.
- При возврате приложения из фона / восстановлении сети — переподписаться и заново вызвать `GET /chat` (WS-соединение могло порваться незаметно).
- При закрытии экрана чата — отписаться от канала (`unsubscribe`), не держать соединение открытым фоново без необходимости.
- Если WebSocket недоступен (сеть блокирует порт, разрыв соединения) — экран должен продолжать работать через чистый REST (отправка/история), просто без мгновенного появления входящих сообщений; на этот случай можно (не обязательно) сделать fallback-поллинг `GET /chat` раз в 10–15 секунд, пока экран открыт.

## 3. UI/UX

- Стандартный экран чата: свои сообщения (`sender == "user"`) — с одной стороны (обычно справа, следуй уже принятому в проекте паттерну выравнивания, если такой чат-компонент уже где-то есть), сообщения администратора (`sender == "admin"`) — с другой.
- Время сообщения (`created_at`) — под текстом, в локальном времени пользователя, коротким форматом (только время, если сегодня; дата+время, если раньше).
- Поле ввода внизу экрана + кнопка отправки. Пустой/пробельный текст — кнопка неактивна. Ограничение 5000 символов — не давать печатать больше (или показывать счётчик при приближении к лимиту).
- **Оптимистичная отправка**: при нажатии «отправить» сообщение сразу появляется в списке (со статусом «отправляется»), после успешного ответа сервера — заменяется на подтверждённое (реальные `id`/`created_at`); при ошибке — показать статус ошибки с возможностью повторной отправки, не терять текст.
- Автоскролл вниз при открытии экрана и при получении/отправке нового сообщения.
- Пустое состояние, если истории ещё нет: понятный текст вроде «Напишите нам, если у вас есть вопросы» (текст — на усмотрение, следуй тону остального приложения и i18n-конвенции проекта — ru/tk).
- Точка входа в чат — в существующей навигации приложения (профиль/настройки/отдельная вкладка — по аналогии с тем, куда уже принято класть подобные разделы в этом приложении). Опционально — индикатор непрочитанных ответов администратора на этой точке входа: считается на клиенте как количество сообщений из `GET /chat` с `sender == "admin" && is_read == false` (отдельного эндпоинта счётчика на бэкенде для этого сейчас нет).
- Локализация всех текстов экрана — через существующий в проекте механизм i18n (ru/tk), как и в остальном приложении, без хардкода строк.

## Что важно не сломать / не придумывать лишнего

- Не создавать несколько диалогов/тредов — по ТЗ это один диалог пользователь↔админ на аккаунт, экран всегда работает с единственным потоком сообщений без выбора «с кем» переписываться.
- Не реализовывать отправку сообщений через WebSocket — отправка **всегда** через `POST /chat` (обычный REST), WebSocket только для получения. Пытаться слать `pusher:trigger` от клиента не нужно и не сработает (клиентские события на приватных каналах в этом проекте не задействованы).
- Не добавлять свои поля/статусы сообщений (например «доставлено» отдельно от «прочитано») — в контракте есть только `is_read`, не выдумывать промежуточные статусы, которых нет на бэкенде.
- Не хардкодить dev-хост/порт Reverb в релизной сборке — вынести host/port/scheme/appKey в конфиг окружения приложения (как уже сделано для `base_url` REST API), значения для прода получить отдельно перед релизом.
