# Задача: раздел «Категории» — дерево до 3 уровней, загрузка иконок, сортировка (Laravel)

## Контекст
Нужно реализовать (или привести в соответствие с ТЗ) раздел «Категории» в админке. Требования — из присланного ТЗ, п.4 «Категории и структура»:

1. Категории имеют до **3 уровней**: категория → подкатегория → подподкатегория.
2. На старте предсозданы **4 корневые категории**: «Оптом», «Розница», «Услуги», «Товары».
3. Для **всех уровней** категорий должна быть возможность: загружать иконку, менять иконку, отображать иконку в мобильном приложении.
4. Управление в админке: создание, редактирование, скрытие/активация, сортировка, управление вложенностью до 3 уровней.

Дизайн-референс (экран 6a в `Admin UI.dc.html`, раздел "Категории — управление деревом до 3 уровней") заменяет старый экран, где:
- иконка задавалась текстовым полем "эмодзи" вместо реальной загрузки файла;
- заголовок страницы и кнопка "+ Добавить" визуально наезжали друг на друга;
- не было явного способа менять порядок строк или скрывать/активировать категорию одним действием.

Перед началом изучи существующий код (миграции/модели/контроллеры/вьюхи), если раздел уже частично реализован — приведи его к паттернам проекта (Blade/Livewire/Inertia/Vue, стиль контроллеров и валидации), не изобретай новый подход.

## Модель данных
Таблица `categories` (self-referencing, если ещё не существует):
- `id`
- `parent_id` — FK на `categories.id`, nullable, `onDelete('cascade')` (удаление родителя удаляет всё поддерево)
- `name_ru` — строка, обязательное
- `name_tk` — строка, необязательное (но желательно для мобильного приложения)
- `icon_path` — строка (путь/URL к загруженному файлу иконки), nullable
- `is_active` — boolean, default `true`
- `sort_order` — integer, default 0 (порядок среди siblings с одним `parent_id`)
- `level` — tinyint (1/2/3), можно вычислять на лету по `parent_id`, но хранить проще для запросов и для enforced-ограничения на 3 уровня
- timestamps

Валидация на уровне модели/сервиса: **запретить создание категории с `parent_id`, у которой `level == 3`** — это и есть ограничение "до 3 уровней" из ТЗ.

Сидер/миграция данных: при первом деплое создать 4 корневые записи (`level=1`, `parent_id=null`): «Оптом» (`Optom`), «Розница» (`Bölekleýin`), «Услуги» (`Hyzmatlar`), «Товары» (`Harytlar`) — не создавать их повторно, если раздел уже используется в проде (проверить по `name_ru` перед сидированием).

## Backend
- CRUD роуты для `categories` (store/update/destroy), плюс отдельные экшены:
  - `PATCH /categories/{id}/toggle-active` — переключение `is_active`.
  - `PATCH /categories/{id}/reorder` — принимает направление (`up`/`down`) или новый `sort_order`, меняет местами `sort_order` только среди siblings того же `parent_id`.
  - `POST /categories/{id}/icon` (или в составе update, `multipart/form-data`) — загрузка иконки: принять PNG/SVG/JPEG/WebP, ограничение размера (например 1 МБ), сохранить в storage (диск, который уже используется в проекте для похожих загрузок — например тот же, что для изображений новостей), вернуть публичный URL.
  - Удаление иконки — отдельный экшен или `icon_path = null` через update.
- При создании/редактировании: `parent_id` может быть пустым (создаёт `level=1`) или ссылаться на существующую категорию `level` 1 или 2 (никогда на `level=3` — вернуть 422).
- При удалении категории с детьми — предупредить на фронтенде (см. UI), backend просто каскадно удаляет через FK.
- В API-ответе для мобильного приложения отдавать дерево категорий (или плоский список с `parent_id`) **с полем иконки** (полный URL) и только `is_active = true` записи — скрытые категории в приложении не показываются.

## UI/UX (страница «Категории»)
Таблица-дерево с раскрытием/сворачиванием веток, колонки: `#` (сквозной номер), «Название» (с отступом по уровню вложенности + шеврон раскрытия у родительских строк + миниатюра иконки 28×28 или плейсхолдер, если иконка не загружена), «Туркменский», «Уровень» (бейдж L1/L2/L3, разные акцентные цвета по уровню), «Статус» (бейдж «Активна»/«Скрыта»), «Действия».

Действия в строке (иконки, справа):
- переключатель видимости (глаз/перечёркнутый глаз) — сразу переключает `is_active`, без модалки;
- «+» — добавить подкатегорию (**скрыта/неактивна для строк уровня 3** — дальше некуда);
- редактировать (карандаш) — открывает панель редактирования;
- удалить (корзина, красный акцент) — подтверждение, текст предупреждения меняется, если у категории есть дети ("вместе со всеми подкатегориями").

Сортировка — две маленькие стрелки (вверх/вниз) слева от номера строки, меняют местами `sort_order` только в пределах своего родителя; неактивны (приглушены), если строка первая/последняя среди siblings.

Кнопка «+ Добавить категорию» — в шапке страницы, справа от заголовка, с нормальным отступом (не наезжает на заголовок).

**Панель создания/редактирования** (выезжает справа, поверх затемнённого фона, как остальные формы в админке):
- «Родительская категория» — select. Пункты: «— Без родителя (уровень 1)» + все категории уровня 1 и 2 (уровень 3 никогда не предлагается как родитель), с визуальным отступом (`—`) по вложенности.
- «Название (рус)» — обязательное текстовое поле.
- «Название (тур)» — текстовое поле, необязательное.
- «Иконка» — **реальная загрузка файла** (не текстовое поле для эмодзи): дропзона/превью 64×64 со скруглением, кнопки «Загрузить»/«Заменить» и «Удалить» (последняя — только если иконка уже выбрана). Подпись поясняет: PNG/SVG, до 1 МБ, квадратная, «показывается в списке админки и в мобильном приложении».
- «Активна» — тумблер с пояснением, что скрытые категории не показываются в приложении.
- Футер панели: «Отмена» / «Создать» (или «Сохранить» в режиме редактирования, кнопка неактивна, пока не заполнено «Название (рус)»).

## Стиль
Тёмная тема, консистентная с остальной админкой: тёмный фон, карточки чуть светлее фона, акцент — индиго/фиолетовый (`#6D63F2` / `#4F46E5` в зависимости от темы проекта), скругления 8–14px, шрифт Golos Text. Используй существующие в проекте цветовые токены/переменные темы вместо новых значений. Полный визуальный референс — секция 6a в `Admin UI.dc.html`.

## Что важно не сломать
- Существующие ссылки категорий с объявлений/товаров (если категория уже используется в других сущностях) не должны ломаться при изменении структуры — проверь внешние ключи на `categories.id` в остальной части схемы перед миграцией.
- Ограничение на 3 уровня должно соблюдаться и на backend (не полагаться только на то, что фронтенд не покажет кнопку «+» у L3) — валидировать `parent_id` на сервере.
- Скрытие (`is_active=false`) родителя не должно автоматически менять `is_active` детей в БД — фильтрация "не показывать в приложении" должна на выдаче учитывать весь путь до корня (скрытый родитель ⇒ скрыты и его видимые дети в выдаче для приложения), уточни у меня, если в проекте уже есть своя логика на этот счёт.
